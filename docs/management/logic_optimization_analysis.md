# 実行時間（Fluid Active CPU）削減に向けたロジック最適化の分析（公式記録）

Vercel の画像生成（OGP および縦長シェア画像）における、Node.js から Edge Runtime への移行がもたらすロジックの改善内訳と、コスト面での有用性を整理した公式記録です。

## 1. ロジック処理の変更点

### A. 実行エンジンの軽量化 (V8 Isolates への移行)
- **Node.js (現状)**: 完全な Node.js 環境をサーバー上で起動するため、メモリ消費と初期起動時間（Cold Start）が多く、CPU を長時間占有します。
- **Edge Runtime (提案)**: Chrome ブラウザの高速エンジンである V8 Isolates 上で動作します。コンテナの起動プロセスをスキップできるため、実行開始までのレスポンスが極めて速く、CPU 負荷が劇的に低くなります。

### B. レスポンス・キャッシュの永続化
- 縦長シェア画像などの生成結果に `immutable` ヘッダーを付与するロジックを追加します。これにより、一度生成された画像は Vercel の Edge Network (CDN) 上に永続的にキャッシュされ、以降のアクセスで **「ロジック関数そのものを動かさない（CPUを全く消費しない）」** 状態に持ち込みます。

---

## 2. コスト（CPU 実行時間）への有用性

### Fluid Active CPU 消費の削減
今回の課金超過の原因となった「Fluid Active CPU」は、「関数が動いている時間」の合算です。
- **効率化1**: 一つの画像生成にかかる実時間を 1〜2秒 から 数十ミリ秒 単位に短縮します。
- **効率化2**: `Edge Runtime` は Node.js よりも実行コストが非常に安く（または無料枠のカウント方法が有利に設定されており）、今回のような 300% 超過を回避する最大の防波堤となります。

## 3. 懸念されるリスク（検証の必要性）
Edge Runtime への移行には、以下のロジック上の制約が伴います。
- **標準ライブラリの制限**: `qrcode` ライブラリ（Node.js 依存）が Edge では動作しません。これを `qrcode-generator` などのピュアな JavaScript 実装に差し替える必要があります。
- **フォント資産の扱い**: 日本語フォントなどの動的読み込みが Edge のメモリ制限（通常 1〜4MB 程度）に収まるよう、ロジックを最適化する必要があります。

---

## 結論
この移行は、単なる修正ではなく **「システムのエネルギー効率を10倍以上に高める」** 抜本的なロジック改修です。Pro プランにアップグレードされた今であっても、将来的なさらなるトラフィック増大に対する「保険」として極めて高い有用性があります。
