# 9coma プロジェクト監督レポート (初期分析)

## プロジェクトの現状分析
`9coma` は、Next.js と Firebase を活用した「自分を構成する9冊のマンガ」をシェアするための非常に洗練されたアプリケーションです。

### 技術的な強み
- **Next.js 14 (App Router)**: 最新のフレームワークを採用しており、拡張性が高いです。
- **Firestore & Caching**: `src/lib/list.ts` において、ISBNベースのキャッシュ機構（`manga_cache`）と並列取得（`Promise.all`）を組み合わせ、効率的なデータ「ハイドレーション」を実現しています。
- **動的OGP**: `src/app/list/[id]/opengraph-image.tsx` により、ユーザーごとのリストを美しくシェアできる仕組みが整っています。
- **UX配慮**:
  - デバウンス処理（500ms）によるスムーズな検索。
  - ドラッグ＆ドロップによる直感的な配置替え。
  - 中央スロット（5番目）のハイライトによる視覚的なアクセント。

## 監督者としての改善提案
現状でも非常に完成度が高いですが、さらに「プレミアム」な体験にするための改善ポイントを挙げます。

### 1. UI/UX の洗練
- **モバイル対応の強化**: モバイル端末でのドラッグ＆ドロップの操作感を確認し、必要であれば長押し（Long Press）での移動や、矢印ボタンによる入れ替えオプションを検討できます。
- **検索結果のフィードバック**: 検索中にスケルトンスクリーンが表示されるのは良いですが、結果がない場合の「空の状態（Empty State）」のデザインをより親しみやすいものにできます。
- **入力バリデーション**: リスト保存時の名前入力や、タイトルの省略表示などの細かいポリッシュ。

### 2. コード品質とメンテナンス性
- **エラーハンドリング**: APIリクエスト失敗時の `alert` を、よりモダンなトースト通知（例: `react-hot-toast` や `sonner`）にアップグレードすることを検討できます。
- **共通コンポーネントの抽出**: 現在 `HomeClient.tsx` が大きくなっているため、スロット部分や検索モーダルを独立したコンポーネントに分割することで、見通しが良くなります。

### 3. SEO & シェア機能
- **メタデータの最適化**: 現状でも実装されていますが、テーマに応じた動的な記述（Description）の強化。

## 🚩 フェーズ更新履歴と現在の詳細仕様 (2026-03-30 時点)

監督として、大規模改修（Phase 52）を経て確立された現在のシステム仕様をここに記録します。これは次期作業者が「一撃で」状況を把握するための、公式の引き継ぎ情報です。

### 1. 検索・外部連携仕様 (Search & API)
*   **利用 API**: 楽天ブックス書籍検索 API (`BooksBook/Search/20170404`)
*   **必須パラメータ**:
    - `booksGenreId: '001001'`: 漫画ジャンルに限定。
    - `outOfStockFlag: '1'`: 在庫なし・絶版商品もヒットさせるために必須（Phase 52で復元）。
    - `formatVersion: '2'`: レスポンスの扱いやすさを考慮。
*   **認証・制限回避**:
    - 楽天APIのドメイン制限を回避するため、リクエストヘッダーに `Referer: 'https://9coma.com'` および `Origin: 'https://9coma.com'` を強制付与（`src/app/api/search/route.ts`）。
*   **無限スクロール (Infinite Scroll)**:
    - `HomeClient.tsx` の検索結果グリッド最下部に `sentinelRef` 要素を配置。
    - `IntersectionObserver` の `rootMargin: '300px'` 設定により、ユーザーが最下部に到達する前に次ページを先読み取得。
*   **ソート機能**:
    - APIの `sort` パラメータ (`standard` / `+releaseDate`) を、UI上の「新着 / 古い」トグルで制御。

### 2. データ構造と最適化 (Architecture & Optimization)
*   **ハイドレーション・スキップ (Performance Optimization)**:
    - Firestore の `lists` ドキュメント保存時に、各スロットの `MangaItem` フルメタデータを冗長的に埋め込む（`api/list/route.ts`）。
    - これにより、個別リスト閲覧時の外部 API 通信をゼロ化し、表示速度の劇的向上と取得コスト削減を実現。
*   **著者スラッグ管理 (`author_slugs`)**:
    - 表記揺れ（スペースの有無など）を吸収するため、全角・半角スペースを除去したスラッグ配列を索引として保持。

### 3. デプロイ・ビルド要件 (Rules & Build)
*   **実行環境**: Node.js 20.x (Vercel)
*   **ビルド制約 (ESLint)**:
    - 本番ビルド時に `no-explicit-any` が強制される。APIレスポンスのパースには `interface RakutenBookItem` 等による厳密な型定義が必須。
*   **メタデータ**:
    - 著者ページ、個別リストページそれぞれに動的な OGP 生成および Web Share API 連携を実装済み。

### 4. 収益化・審査対策 (Monetization & Compliance)

## 📌 監督からの引き継ぎ事項 (Handover Items)

1.  **検索パラメータの「絶対死守」**: 今後 API ロジックを触る際は、決して `outOfStockFlag` と `Referer` を外さないこと。これがないと「検索結果が半減する」または「403エラーで全停止する」という致命的な退化を招く。
2.  **型定義の徹底**: 「作業が早いが雑な AI」が `any` を使いがちだが、Vercel でのビルドが 100% 失敗するため、必ず型定義を先行させること。
3.  **環境変数とインフラの整合性**: AdSense ID 等を `layout.tsx` で参照する際、ローカルの `.env.local` だけでなく、**Vercel などのホスティング先ダッシュボードにおいて「Environment Variables」として値が正しく設定されているか** を必ず確認すること。設定が漏れると本番サイトでタグが空になる。ハードコードによる「その場しのぎ」は厳禁である。
4.  **リスト保存時のデータ完全性**: ハイドレーションをスキップしているため、保存時に `title` や `imageUrl` が欠けた状態で Firestore に書き込まれると、閲覧ページで一生修正不能な不具合となる。POST 時のバリデーションは常にシビアに行うこと。
5.  **デザインのアイデンティティ**: 黄色を基調としたポップなデザインと、3×3 グリッドのドラッグ＆ドロップ、および中央スロットの強調表示は、このサービスの核である。これを損なう改変は「改悪」とみなす。
