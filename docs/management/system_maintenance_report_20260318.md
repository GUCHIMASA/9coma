# システム保守点検 & 負荷想定評価レポート (2026-03-18)

直近のデプロイ（著者ページ強化、一覧ページ改善）を経た現在のシステム状況を、保守・運用面から点検いたしました。

## 1. サーバー負荷・スケーラビリティ評価
### [判定: 非常に安定 / 低負荷]

- **ISR (Incremental Static Regeneration) の効果**:
  著者ページや一覧ページに `revalidate = 3600` を設定しているため、同一ページへのアクセスが急増（バズ等）しても、Firestore への問い合わせは「1時間に1回」に抑制されます。数万〜数十万 PV/時 レベルのスパイクにも、現状の構成で十分耐えられます。
- **OGP 動的生成**:
  Vercel のインフラ上で画像キャッシュが効くため、サーバーリソースの消費は最小限に抑えられています。

## 2. データベース (Firestore) コスト評価
### [判定: 最適化済み / 健全]

- **クエリの効率化**:
  `array-contains` インデックスを活用した著者検索により、全件スキャンのような高コストな処理を回避しています。
- **データヒーリング (Hydration)**:
  古いリストの欠損データを楽天 API から補完するロジックは、表示を正常に保つために有効です。一度補完されれば `manga_cache` に永続化されるため、同じリストへの二度目以降のアクセスは高速かつ低コストです。

## 3. 注意点と将来へのアドバイス (シビアな視点)
### A. 人気著者のカウント上限
現在の `getSelectionCountByAuthor` は `limit(500)` を設けています。特定の著者が 500 リスト以上に含まれるような超大型ヒットとなった場合、カウントが 500 件分で頭打ちになる可能性があります。
- **対策**: 将来的に PV が大幅に伸びた際は、著者ごとに `total_count` カラムを持つサマリー用ドキュメントを導入することで、処理をさらに軽量化（読取 1 件のみ）できます。

### B. 楽天 API レートリミット
大量の「過去の不完全なリスト」が一斉にアクセスされた場合、楽天 API の秒間リクエスト上限に触れるリスクが僅かにあります。
- **対策**: 現状は `manga_cache` が防波堤になっていますが、トラフィックが 10倍以上に成長した際は、API 呼び出しのキューイングや、エラー時の指数バックオフの強化を検討してください。

## 結論
現状、システムは**「スマートかつ経済的」**に設計されており、特段の懸念事項はありません。次のフェーズ（広告収益化の本格運用やユーザー投稿の大規模増加）に進む準備は十分に整っています。
